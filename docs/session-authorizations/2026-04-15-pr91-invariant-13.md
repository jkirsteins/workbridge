# Authorization: invariant 13 "Fresh Claude session per stage" - PR #91

## Metadata

- **Date:** 2026-04-15
- **Pull request:** https://github.com/jkirsteins/workbridge/pull/91
- **Work item branch:**
  `janis.kirsteins/remember-claude-session-id-for-each-workitem-b214`
- **File touched:** `docs/invariants.md`
- **Invariant touched:** "13. Fresh Claude session per stage"

## Bullets touched

The entire body of invariant 13 in `docs/invariants.md`, specifically:

- The new paragraph beginning "**'Fresh' means per `(WorkItemId,
  WorkItemStatus)`, not per workbridge process.**" that introduces
  the deterministic UUID v5 scheme, explains the `--resume` /
  `--session-id` dispatch, and notes the scheme is pure (nothing
  persisted in the workbridge data model).
- The new paragraph explaining that stage transitions still produce
  new sessions because `(wi_id, Implementing)` and `(wi_id, Review)`
  yield different UUIDs, and that cross-stage context bleed remains
  impossible by construction rather than by process isolation.
- The revised "Session lifecycle per stage" bullet list, which now
  describes the Implementing and Review entries as "either continues
  the `(wi_id, stage)` transcript (on restart) or creates a new one
  under that tuple's deterministic UUID" instead of "a fresh session
  starts".
- The rewritten sentence that used to read "Sessions gain context
  only via MCP tools ... not from prior sessions", now reading
  "Sessions gain conversational context only from the deterministic
  transcript for the same `(wi_id, stage)` tuple and from MCP tools"
  and then explicitly restating that they do NOT gain context from
  sessions belonging to a different stage or a different work item.
- The closing paragraph confirming that the review gate's ephemeral
  `claude --print` subprocess and the global assistant drawer are
  explicitly outside the scheme.

## User rationale (verbatim)

From the work item description that drove PR #91:

> If we quit workbridge and resume, it should re-spawn the last
> known claude sessions using their session IDs. There should be 1
> session to resume, so when we switch phases, we should update it.

The user explicitly wanted restarting workbridge to preserve
Claude's in-stage conversational context for active work items,
while still keeping the "switch phases -> new session" semantic of
invariant 13. The original invariant text was strict enough that
any implementation of this feature technically violated it, because
it literally said "not from prior sessions". The spirit of the rule
(cross-stage isolation, plan as the handoff contract) is still
honoured; only the letter needed a carve-out for the restart-resume
case within the same `(wi_id, stage)` tuple.

## Scope boundary

This authorization covers ONLY the invariant 13 edits listed in
"Bullets touched" above. It specifically does NOT authorize:

- Any change to other invariants in `docs/invariants.md`. Invariants
  1-12 and 14-15 remain untouched on this branch.
- Relaxation of the "fresh session per stage transition" rule for
  anything other than the restart-resume case within the same
  `(WorkItemId, WorkItemStatus)` tuple. Stage transitions (Planning
  -> Implementing -> Review) still produce a new deterministic UUID
  because the tuple changes, so the Review session is structurally
  unable to see the Implementing transcript. Cross-stage context
  bleed remains impossible by construction.
- Any change to the review gate's ephemeral `claude --print`
  subprocess or the global assistant drawer. Both remain explicitly
  outside the deterministic UUID scheme; they are one-shot or
  separate-scope sessions and must not share identity with
  work-item stage sessions.
- Any relaxation of the session-keyed-by-`(WorkItemId,
  WorkItemStatus)` HashMap enforcement in the code. The sessions map
  still refuses to reuse a key across stage transitions, and
  `apply_stage_change` still cancels pending session-opens for the
  old stage before the background worker can deliver a stale result.
- Any change to the "manual advance from Planning to Implementing
  is blocked - the plan must be set via MCP" rule. That constraint
  is preserved verbatim in the new text.

An adversarial reviewer reading the diff against
`docs/invariants.md` should confirm that every edited line falls

## Amendment 1: persist a stage-transition counter nonce

### Date and channel

- **Date:** 2026-04-15 (same review session, later follow-up round)
- **Authorization channel:** Codex adversarial review flagged that
  the original `(WorkItemId, WorkItemStatus)` derivation collapsed
  repeated visits to the same stage, so a cycle like
  `Planning -> Implementing -> Blocked -> Planning` or a
  `Review -> Implementing` rework would resume the prior
  transcript instead of starting a fresh session - breaking the
  plan-handoff guarantee the invariant was meant to enforce. The
  reviewer ("Claude Code") presented three options to the user:
  - **A.** Persist a stage-transition nonce in `WorkItemRecord`
    and fold it into the deterministic UUID derivation.
  - **B.** Accept cycling-reuse as a known limitation.
  - **C.** Revert the resume feature entirely.

  The user replied verbatim:

  > let's do option A

### What this amendment authorizes

The user explicitly approved Option A, which requires two
additional edits beyond the "Bullets touched" list above:

1. **Additional `docs/invariants.md` edits inside invariant 13's
   body:** replace the "(WorkItemId, WorkItemStatus)" framing with
   "(WorkItemId, WorkItemStatus, stage_transition_count)" in every
   place the UUID derivation is described, and remove the claim
   that "nothing is persisted in the workbridge data model". The
   counter IS persisted on `WorkItemRecord`, so the literal claim
   is no longer true. The invariant body must explain what the
   counter is, why it exists (cycle-back correctness), how it is
   bumped (`LocalFileBackend::update_status` on every real
   transition, skip on no-op), and what the migration story is
   (serde default `0` for records written before the field
   existed).
2. **A new field on `WorkItemRecord`:** the counter lives as a
   `stage_transition_count: u64` field with
   `#[serde(default)]`. `LocalFileBackend::update_status` bumps
   it on every real status change and leaves it alone on no-op
   writes. `assembly.rs` passes it through to `WorkItem`. This
   is a new persistence surface that the original authorization
   did NOT cover, and every call site that constructs a
   `WorkItemRecord` or a `WorkItem` must initialise the field
   (tests use `0` because they run on fresh synthetic records).

### Scope boundary (still holds)

Everything in the original "Scope boundary" section above still
applies. This amendment does NOT relax:

- Cross-stage isolation. The counter makes cross-stage isolation
  STRONGER, not weaker, because cycling back to the same stage now
  lands on a different UUID.
- The review gate / global assistant carve-out. Both remain
  outside the deterministic UUID scheme entirely; neither reads
  `stage_transition_count`.
- Any other invariant. Invariants 1-12 and 14-15 remain
  byte-identical on disk. The only `docs/invariants.md` edit
  anywhere on this branch is inside invariant 13's body, as
  described in "Bullets touched" above plus the counter-related
  edits described in this amendment.
- The MCP-only-context rule for everything EXCEPT the same
  `(wi_id, stage, count)` tuple's own transcript. A session may
  resume ITS OWN prior transcript on a workbridge restart while
  the counter is unchanged; it may NOT inherit context from a
  different tuple.
