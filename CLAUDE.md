# Workbridge Development Rules

Read and follow CONTRIBUTING.md before making changes.

## Review Policy

### How severity overrides interact with session authorizations

The rules in this section are **default-overridable** unless a specific rule is tagged `[ABSOLUTE]`. In the adversarial review loop, every default-overridable rule applies as stated **unless the specific change is covered by an explicit user authorization recorded in the session's Authorizations file** (see the `claude-adversarial-loop` skill's "Authorizations File" section). A valid authorization must name the exact scope (file path, symbol, or change description) and the user's rationale; blanket "do whatever" authorizations are not sufficient.

This gives the reviewer a clean suppression path when the user has explicitly sanctioned a one-off exception, while keeping the rules strict for every unauthorized change. The default (no authorization present) is the same stance as before: findings in these categories are P0.

### Severity overrides

- Using custom widgets when a ratatui built-in widget (from `ratatui-widgets` or `ratatui-core`) could accomplish the same thing is P0 unless the specific widget is covered by a session authorization naming the file and the reason a built-in is insufficient.
- Reimplementing layout, rendering, or styling logic already provided by ratatui is P0 unless the specific reimplementation is covered by a session authorization.
- Modifying docs/invariants.md is P0 unless the exact edit is covered by a session authorization that names the specific line/bullet being changed, the user's rationale, and the scope boundary. Default interpretation is "no edit"; vague or blanket authorizations do not count. Each authorization must be specific enough that a reviewer reading only the authorization text can verify the edit matches it.
- Code changes to a system described in docs/ without corresponding docs updates is P0 unless the missing doc update is explicitly authorized (e.g., user says "docs update will follow in a separate PR, skip the doc-parity check for this one").
- Code that touches harness invocation (any spawn of an LLM coding CLI, any change to the argv/system-prompt/MCP-injection/tool-allowlist/stage-reminder path, any change to `Session`/PTY lifecycle, any new headless spawn) must update `docs/harness-contract.md` in the same PR. P0 unless the missing doc update is explicitly authorized. The three known spawn sites are enumerated in the "Known Spawn Sites" table of that doc; the table and the per-clause Implementation Map must both stay in sync with the code. Default interpretation is "no harness invocation change ships without a contract update".
- Code changes to the CLI contract must update `docs/cli.md` in the same PR. P0 unless the missing doc update is explicitly authorized. The CLI contract is the complete user-facing surface dispatched from `src/main.rs::handle_cli` and its delegates, plus the `--mcp-bridge` entry point handled earlier in `main()`. A CLI contract change is any of: adding a new top-level subcommand; adding, removing, or renaming a subcommand argument or flag; changing the set of supported `config set` keys or the validation of their values; changing which subcommand invocations exit 0 vs 1; changing printed-usage strings such that they document a different argv shape; changing the stdout/stderr format of any subcommand in a way a script could observe. The covered surface today is `workbridge` (TUI launch), `workbridge repos` (list / add / add-base / remove, including the `--all` flag on `list`), `workbridge mcp` (add / remove / list / import, including every flag on `mcp add`), `workbridge config` (show + `set`, including the supported-keys list), `workbridge seed-dashboard`, and `workbridge --mcp-bridge --socket <path>`. Default interpretation is "no CLI surface change ships without a cli.md update". This rule is a specialization of the general "code changes to a system described in docs/ without corresponding docs updates is P0" rule - it exists as its own bullet because the CLI surface is spread across several handler functions in `src/main.rs` and is easy to regress without noticing that the docs drifted.
- Adding a new harness adapter (a second LLM CLI alongside the existing `claude` reference) is P0 unless every clause C1..C13 in `docs/harness-contract.md` gets a new column entry for that adapter marked `supported` / `workaround` / `not supported` with a one-paragraph justification, AND the new adapter's spawn sites are added to the Known Spawn Sites table. Default interpretation is "an adapter that does not document itself against every clause is incomplete and must not land".
- Relaxing linter rules, suppressing warnings, or skipping git hooks (--no-verify) is P0 unless the specific suppression is authorized. Each `#[allow(...)]`, each suppressed warning, or each `--no-verify` use needs its own authorization naming the exact rule being relaxed and the exact scope.
- Underscore-prefixing struct fields or variables to hide dead code warnings instead of removing the dead code is P0 unless authorized (e.g., the user has a staged rollout where a field is added before its reader).
- Adding redundant filtering (a second filter for the same constraint that's already enforced at the source, e.g., `gh` CLI `--author @me`) is P0 unless the duplicate filter is explicitly authorized as defense-in-depth for a specific threat model.
- Adding new external API calls (gh api, network requests) without verifying the information is not already available from existing calls or CLI flags is P0 unless the new call is authorized with evidence that the existing data isn't sufficient.
- **[ABSOLUTE]** Performing blocking I/O (git commands, network requests, slow file operations) on the main UI thread instead of spawning a background thread is always P0 and cannot be overridden by session authorization - this is a correctness invariant, not a preference. A blocking call freezes the UI for real users and there is no legitimate "authorized exception" for that. See docs/UI.md "Blocking I/O Prohibition". This includes indirect I/O: a trait method like `worktree_service.github_remote(...)` or `backend.list()` can shell out to `git`/`gh` or hit the filesystem under the hood, so reviewing a call site alone is not enough - you must check the implementation of every method called on a UI-thread code path (timer tick, event handler, reassembly, reconstruction, render) and confirm it is either a pure in-memory read or explicitly documented as non-blocking. When in doubt, prefer values already cached on `App` (e.g. `self.repo_data[path].github_remote` instead of `self.worktree_service.github_remote(path)`).
- Prefer structural ownership over manual correlation: at least P1, default-overridable. State that has to be matched back to its owner via if-checks at every cleanup site is fragile - it's easy to forget a check, easy to clobber an unrelated entry on retreat / delete / cancel paths, and the bug stays dormant until two instances of the state coexist. State that lives inside a `HashMap<Owner, _>` entry, a struct field on the owner itself, or any other container that already encodes ownership is robust because dropping the owner drops its state automatically. Flag patterns where the same identifier appears across multiple sibling fields that all have to stay in sync (e.g. `Option<Receiver>` + `Option<ActivityId>` + `Option<WorkItemId>` triplets), or where a singleton `Option<Receiver>` is filtered by an `if owner_matches` guard at every read / cleanup site. The fix is almost always to move the state into a map keyed by the owner, or onto the owner struct directly. Severity escalates to P0 if the lack of structural ownership is already causing observable cross-item state corruption.
- User-initiated event handlers that trigger remote I/O (`gh`, network, `git fetch` / `push` / `pull`) must go through `App::try_begin_user_action` is P0 unless the specific bypass is covered by a session authorization naming the handler function and the reason the helper is insufficient. The helper lives in `src/app.rs` and is documented in `docs/UI.md` "User action guard". Default interpretation is "every user-initiated remote-I/O spawn belongs to one of the `UserActionKey` variants (`GithubRefresh`, `PrCreate`, `PrMerge`, `ReviewSubmit`, `WorktreeCreate`, `UnlinkedCleanup`, `DeleteCleanup`) and gets its single-flight / debounce admission from the helper." Bypassing the helper to shell out or spawn a thread directly from an event handler reintroduces the `Option<Receiver>` + sibling `Option<ActivityId>` anti-pattern the helper was built to eliminate, and is the exact failure mode behind the rapid-Ctrl+R fetcher dog-pile bug. This rule is a policy preference (hence default-overridable, not `[ABSOLUTE]`) - it is NOT a correctness invariant like blocking-I/O; if a caller genuinely needs per-item keying or a custom queue the authorization must name the caller, the reason, and the scope.
- Writing, patching, or otherwise mutating files outside of workbridge's own owned data / temp directories in order to influence the behavior of a third-party tool is at least P1, default-overridable. Examples: editing `~/.claude.json` to bypass Claude Code's workspace trust dialog, writing helper configs into `~/.config/`, dropping override files into a git worktree to change another tool's defaults, or patching a third-party tool's state file to pre-register a flag. File injection hides the fact that the fix depends on undocumented or deliberately locked-down behavior of another tool, creates version-fragility (the third-party tool may change the format or field semantics and silently regress the workaround), and usually indicates the real problem is being papered over instead of solved. The preferred fix is either (a) to use the third-party tool's documented configuration surface, (b) to change the call site so the third-party tool's default behavior is acceptable (e.g. pick a different cwd, spawn a different subcommand), or (c) to accept the tool's actual behavior. Severity escalates to P0 if the injection would flip a security-relevant field the user has explicitly left in its default state (e.g. `hasTrustDialogAccepted`, permission modes, disabled-safety flags). The motivating rejection for this rule was a proposal to read-modify-write `~/.claude.json` to pre-register trust for a workbridge-owned scratch directory; the cleaner fix was to spawn the global assistant in a non-home scratch cwd and let Claude Code's own trust-persistence mechanism handle the first prompt.
- Introducing files into a worktree that are not meant to be tracked in git is at least P1, default-overridable. Any file whose purpose is environmental rather than part of the repository's source tree must not be added, modified, or committed as part of task work. This covers harness/runtime configuration (e.g. `.mcp.json`, `.claude/settings.local.json`), local IDE/editor state (e.g. `.vscode/settings.json`, `.idea/`), machine-specific overrides, secret/env files (e.g. `.env`, `.env.local`), build/test artifacts, scratch files, and anything else that would normally live only in a contributor's local checkout. The litmus test is: "would the project's maintainers expect every contributor to have this exact file checked in, and would they review it as part of normal source review?" If the answer is no, the file does not belong in a feature branch, PR, or merge - it leaks local state and pollutes history. A `.gitignore` entry covering the path is strong evidence the file is in this category, but absence from `.gitignore` is not a free pass: the test is about intent, not tooling state. Flag any diff that introduces or edits such a file unless the specific path is covered by a session authorization that names it and the reason it legitimately belongs in the repo.
- Making a user-facing claim (toast text, label text, status bar message, modal body, alert message, context bar, or any other rendered string that asserts application state) that the code has local information to verify but doesn't check is P0 unless the specific claim is covered by a session authorization naming the call site and the reason the check cannot be wired up. The canonical failure mode: a helper returns a success/failure signal (bool, `Result`, error enum, `Option`) and the UI pushes an unconditional "it happened" message while ignoring that signal. Concrete example: calling `clipboard::copy` - which returns a bool indicating whether any backend actually wrote the value - and pushing `Copied: <value>` without branching on the result. When the clipboard backends all fail, the toast lies: the user reads "Copied" and pastes stale content later. The rule is structural - if the signal is reachable at the call site via a method call, error branch, or return value, the rendered text MUST reflect the actual outcome (success, failure, or a kind-specific middle state). Severity downgrades to P1 (not P0) only when the outcome is **genuinely unknowable from the code's vantage point** - for example OSC 52 writes bytes to stdout and there is no cross-terminal way to detect whether the receiving terminal actually pushed them to the clipboard. In that case the honest UI must express uncertainty ("Copy requested..."), hedge the wording, or drop the claim entirely, but the rule stays P1 rather than P0 because no ground-truth signal exists to consult. "Too hard to wire up" or "the check requires refactoring the helper" is NOT unknowable - if the signal is reachable, the claim must honour it. Lying to users about application state is a bug, not a style preference.
- TUI text inputs must support paste. Adding a new text-input field (`rat_widget::text_input::TextInputState` or `rat_widget::textarea::TextAreaState`, or any future equivalent) without wiring it into `event::handle_paste`'s modal routing (`route_paste_to_modal_input` in `src/event.rs`), or removing paste support from an existing input, is **P1, default-overridable**. Each exception needs a session authorization naming the specific field and rationale (for example, an input where multi-line paste would be actively harmful and the omission is intentional). The implementation pattern - single-line inputs pass through `flatten_paste_for_single_line` to strip newlines, multi-line `TextAreaState` receives the payload verbatim, no-text-input focus arms return `false` so the paste is swallowed without leaking to the PTY - is documented in `docs/UI.md` "Paste handling". This is P1 rather than P0 because it is a UX consistency rule, not a correctness invariant; legitimate exceptions exist (e.g. a numeric-only field where paste would routinely produce invalid data).
- **[ABSOLUTE]** Session titles (right-panel tab title, dead-session placeholder, Ctrl+\\ toggle hint, or any other rendered string that names which harness is running) MUST be downstream of live harness state, never upstream-defaulted to a vendor name. If no harness is committed for the current context (no `harness_choice` entry, no global-assistant harness configured), the UI MUST render a neutral placeholder (`App::SESSION_TITLE_NONE` = `"Session"`) - NOT the hardcoded reference-implementation name ("Claude Code"). This rule is `[ABSOLUTE]` because showing "Claude Code" in the tab title while the user has picked Codex (or has picked nothing and no session runs) is a user-facing claim about state that the code can locally verify but currently lies about - the same failure mode as the "user-facing claim ignores available signal" rule above, scoped to the harness-identity axis. Canonical failure: `App::agent_backend_display_name` falling through to `self.agent_backend.kind().display_name()` as a last resort, where `self.agent_backend` is hardcoded `ClaudeCodeBackend`. The resolution order is (1) per-work-item `harness_choice` for the selected item, (2) global-assistant harness if the drawer is open, (3) neutral placeholder. Never fall through to the static `self.agent_backend`. Harness adapters MUST NOT influence the UI title outside the live-session context; adding a new adapter does not change this rule. See `docs/UI.md` "Session tab title" and `docs/harness-contract.md` for the architectural principle.
- **[ABSOLUTE]** Silent fallbacks to a default harness (or to any other "hidden default" when the user's intent is unknown or unresolvable) are always a P0 violation and cannot be overridden by session authorization. On error - missing harness choice, unreachable binary, unparseable config, any ambiguity about which LLM CLI should run - the code must surface an explicit error to the user (toast, modal, status message, or a blocked stage) and must NOT proceed with a substituted default. Canonical failure mode: a spawn site reads a per-work-item `harness_choice` and resolves it via `.unwrap_or_else(|| Arc::clone(&self.agent_backend))` or equivalent - the user had previously picked Codex, the map was empty after restart, and the code silently ran Claude against their code instead. This rule is broader than harness selection but harness selection is the motivating case: any place where the user's explicit choice is missing and the code could either abort-with-error or fall-back-to-a-default, it MUST abort with an error. The user gets to decide what to do next; the code does not get to decide for them. The three known harness spawn paths (work-item interactive respawn via `spawn_session` / `finish_session_open`, review gate via `spawn_review_gate`, rebase gate via `spawn_rebase_gate`) must all enforce this rule symmetrically - if one path aborts and another silently substitutes, the asymmetry itself is the bug. This rule is `[ABSOLUTE]` because the harm (wrong LLM provider reaching into the user's code, wrong billing, wrong tool affordances, potentially wrong trust boundary) is not recoverable after the fact and there is no legitimate "authorized exception" for running the wrong agent against the user's repo.
- Tests that cause side effects on the host environment outside `std::env::temp_dir()` are P0 unless the specific side effect is covered by a session authorization naming the test and the rationale. Covered side-effect channels include: the system clipboard (via `arboard`, OSC 52, or any other path), `directories::ProjectDirs` / `BaseDirs` / `UserDirs` paths, `$HOME` writes, environment-variable mutations without deterministic restore, persistent files or sockets outside `std::env::temp_dir()`, spawned processes that outlive the test, notification / audio / visual APIs, and terminal escape sequences written to stdout/stderr. The reference anti-pattern was `arboard::Clipboard::new().set_text(...)` running during `cargo test` and clobbering the user's real clipboard with test-fixture strings (`"feat/my-branch"`, `"workbridge"`) that had been registered as copy targets in event-pipeline tests. All side-effect APIs must be reached only through `src/side_effects/`, which is `#[cfg(not(test))]`-gated and returns a no-op / `None` under test. The pre-commit grep check in `hooks/pre-commit` ("side-effect gate check") enforces bypass rejection at commit time - any staged `.rs` file outside `src/side_effects/` that references `arboard::`, `directories::ProjectDirs` / `BaseDirs` / `UserDirs`, `std::env::home_dir`, or `std::env::temp_dir` is rejected. Default interpretation is "any side effect visible outside the test process is a bug". This rule is default-overridable (not `[ABSOLUTE]`) because there are legitimate bounded exceptions - for example, `src/session.rs` spawns short-lived `sleep 60` / `sleep 0` processes to cover the PTY lifecycle, and `src/mcp.rs` binds a UUID-suffixed Unix socket under `/tmp` for the socket-server smoke test. Such exceptions must be documented in `docs/TESTING.md` and the authorization must name the specific test and why the gating helpers are insufficient.

### Review guidelines

- Prefer ratatui's built-in widgets (e.g., `Paragraph`, `Block`, `Table`, `List`, `Tabs`, `Gauge`, `Sparkline`, `BarChart`, `Canvas`, `Scrollbar`) over custom widget implementations whenever possible
- When a ratatui widget almost fits the use case, prefer composing or wrapping the built-in widget rather than building from scratch
- Custom widgets are acceptable only when no ratatui widget (or reasonable composition of widgets) can achieve the required behavior
- docs/invariants.md is immutable - any diff that modifies it must be flagged as a violation regardless of context
- If a change touches a system that has a corresponding doc in docs/ (e.g., aggregation, inbox, worktree management, error states, etc.), the review must verify that the doc is updated to reflect the new behavior. Missing doc updates are P0
- Harness invocation code changes (any new or modified `Command::new` call whose program is an LLM coding CLI, any argv / system-prompt / MCP-injection / tool-allowlist / stage-reminder change, any change to `Session` or PTY lifecycle) must update `docs/harness-contract.md` in the same PR - that doc is the authoritative spec for what any harness adapter (`claude` today, future `codex` / `opencode` later) must provide, and the contract exists so these changes cannot drift silently from the implementation map and reference payloads
- Never relax linter requirements, add #[allow(...)], suppress warnings, or skip git hooks to make code pass CI. Fix the underlying issue instead
- All blocking I/O (git commands, network requests, subprocess calls) must run on a background thread with results polled via crossbeam channels - never on the main UI thread. See docs/UI.md "Blocking I/O Prohibition" for the pattern
- Also reference CONTRIBUTING.md and docs/invariants.md as authoritative project files
